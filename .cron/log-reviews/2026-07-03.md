### ✅ Already fixed (verified): diacritic-slug redirect 500s
`get_dictionary_by_url_or_id` now resolves NFC/NFD slug forms (`903f3276`, 12:59 UTC), which surfaced
a latent bug: the canonical redirect built a `Location` header from the **raw unicode** slug
(`ngəmba`, `mišär-tatar`, `nağaybäk`) → SvelteKit throws "characters that cannot be used in HTTP
headers". `canonical-path.ts` now `encodeURIComponent`s the canonical url (`0068a0d7`, 12:46 UTC;
carries the explicit test *"encodes a legacy non-ascii canonical url instead of crashing"*), and the
`[redirectId]`/`[dictionaryId]` redirects encode their segments too. **Last error 11:15 UTC, deploy
~13:00, zero recurrence** → no action.

### OPEN: the `AccessHandle is closed` retry loop (real)
`classify_sync_failure` (<File path="site/src/lib/db/sync/sync-failure-classify.ts" line="49" />) has no
predicate for OPFS sync-access-handle lifecycle errors, so `"AccessHandle is closed"` falls through to
`other` → `error`, and `other` is **not** in `THROTTLED_KINDS` → all 112 retries shipped as
unthrottled `error` rows (16:35→21:32 UTC, dict `tuscarora`). Two layers:
- **Phase B (classify):** add an OPFS-lifecycle predicate (`/AccessHandle is closed|handle is closed|
  InvalidStateError|the database connection is closing/i`) → return `network` (warn + throttle), so a
  torn-down leader worker can't spam the errors panel. Grounded in today.
- **Deeper (worker):** 112 retries against a **closed** handle suggests the dict-sync engine keeps
  looping instead of stopping/re-electing when the OPFS handle is gone. Worth a look at whether
  `AccessHandle is closed` should abort the sync loop and trigger leader re-election rather than retry.
