Current-source sweep covered client editing/search/audio flows, dict sync, background jobs, and API
failure paths. Existing instrumentation is broad (105 explicit track/log calls in the reviewed route,
worker, and server areas), but several core server failures still only reach ephemeral Docker output:

1. **Media upload presign + serving URL failures** — `routes/api/upload/+server.ts` and
   `routes/api/gcs_serving_url/+server.ts` → propose `log_server_event`, error level, with HTTP status,
   media kind, dictionary/user identifiers (never the signed URL). These are core editor workflows and
   tonight's successful-action counts cannot tell whether failed attempts occurred.
2. **Dictionary creation/catalog/invite failures** — `routes/api/dictionaries/create/+server.ts`,
   `[id]/catalog/+server.ts`, and invite acceptance → `log_server_event`, error level, fields
   `{ user_id, dict_id, stage }`. Current `console.error` disappears across container replacement.
3. **Message compose/reply delivery failures** are already reported to admins in the UI and console,
   but are not durable telemetry. Add one server event per failed send with thread/stage/provider status;
   do not include message content or recipient email.

These are recommendations only. The command forbids implementation, and none is promoted above P3
without a live occurrence.
