### 🟡 P3 — `featured_entries.entry_id` UNIQUE double-insert (new homepage ⭐ toggle)

Root cause **verified in current source** (`src/routes/[dictionaryId]/entry/[entryId]/+page.svelte`,
`toggle_star()`, lines 72–86):

```ts
let star_row_id = $state<string | undefined>(undefined)
$effect(() => { star_row_id = dict_db?.featured_entries.rows.find(r => r.entry_id === page.params.entryId)?.id })
async function toggle_star() {
  if (star_row_id) { await dict_db.featured_entries.delete(star_row_id); return }
  const [last] = await dict_db.featured_entries.query({ order_by: 'sort_key DESC', limit: 1 }).snapshot()
  await dict_db.featured_entries.insert({ entry_id: entry.id, sort_key: key_between(last?.sort_key ?? null, null) })
}
```

`star_row_id` is only refreshed by the `$effect` *after* the insert → sync → live-rows update round-trip.
A rapid second click (the drilled session shows a manager clicking "Feature this entry" 3× in ~3 s)
fires while `star_row_id` is still `undefined` → a **second `insert` on the same `entry_id`**, which the
schema's UNIQUE natural key rejects. Data stays correct (no dup created); it just logs a client error.
7 rows / 1 user / 1 session, all on the current build (08:24 UTC today).

- **Recommend (do NOT apply):** make the star idempotent. Cleanest is an in-flight guard
  (`let starring = $state(false)` around `toggle_star`) *or* an optimistic local set of `star_row_id`
  before the await; the most robust is `INSERT … ON CONFLICT(entry_id) DO NOTHING` at the dict-client
  insert layer so any double-fire (human or agent via `/api/v1`) is naturally idempotent. → action item.

## 5. Coverage gaps (Phase B)

Prioritized by "would it have answered a question I had to infer by hand this run?"

1. **HIGH — `dict_boot` timing (the one unmeasured segment of the homepage→dictionary journey).**
   `src/lib/db/dict-client/dict-instance.ts` walks boot stages `snapshot_fetch → opfs_open → migrate →
   probe` via `report_progress`, but those only **tick the idle watchdog — nothing is logged**
   (verified: the only client `track_timing` in the app is `search`; `page_load` comes from
   `perf.ts`). The cold first-hop-into-a-dictionary (`home → entries` p95 up to 3.7 s) is currently
   **inferred from `navigation` timings**, mixing warm and cold. Emit
   `track_timing({ name: 'dict_boot', duration_ms, context: { cold: boolean, snapshot_bytes, stage_ms } })`
   when the viewer finishes booting. **Why:** directly measures the new homepage's most latency-sensitive
   step, cold-vs-warm, per dictionary — and unblocks the Phase C panel below.
2. **MEDIUM — featured-entry ⭐ toggle has zero analytics + only surfaces on failure.** The featured
   strip is a *centerpiece of the new homepage*, yet star/unstar emits no `track` event, so adoption is
   invisible; today it only appeared in the logs as the UNIQUE error. Add
   `track({ event: 'entry_featured' | 'entry_unfeatured', props: { dictionary_id, entry_id } })` (new
   constants in `log-events.ts` + `ALL_TRACKED_EVENTS`). **Why:** measures whether editors actually use
   the marquee feature, and gives the toggle a success signal alongside the (to-be-fixed) failure path.
3. **LOW — `Internal Error` / `Photo processing error … fetch failed` carry no structured context.**
   The photo-processing 500 and the context-less `Internal Error` (95 rows, 1 user) are hard to triage
   because neither logs a `context.route`/`dictionary_id`/source id. When either is raised server-side,
   route it through `log_server_event` with structured fields. **Why:** low frequency today, but each is
   currently a dead-end when it does fire.
