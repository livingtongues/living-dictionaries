### The `props_invalid_value` correction — a real bug, currently masked by the conlang fence

The 07-15 digest ruled this "admin-3-gated preview only… only Jacob hits it… zero real-user exposure."
**That is now confirmed WRONG on the gating,** and only *accidentally* right on the impact:

- **Who hit it today:** `empleomujerctg@gmail.com` on **`/cormani/grammar`** (2×, Mac) and
  `tryonz@proton.me` on **`/lunvot/grammar`** (1×, Android). **Neither is a site admin.**
- **Why they could reach it:** `GrammarSection.svelte` (L32-35) grants **managers (non-admin-3)** a
  `can_prose_edit` path that opens `SectionEditor` in `prose_only` mode (L122). Grammar editing is
  **not** admin-3-only — the intro-prose editor is open to any dictionary manager.
- **Why it's zero *mission* impact today:** `cormani`, `lunvot`, `rhenic` all carry `bucket:"conlang"` —
  the fenced conlang fork (decisions 07-07: don't count/report on it). So the only people currently
  using the new grammar editor are conlang builders. **The bug itself is real production code**, not a
  preview — it just hasn't been struck by a real-dict manager yet.
- **Root cause (still pinned, verified in current code):** `SectionEditor.svelte:146` / `:159` bind
  `bind:value={draft_body[bcp]}` / `draft_usage[bcp]` — `undefined` for a locale the section has no
  body/usage in — into `MarkdownEditor`'s `value = $bindable('')` (`MarkdownEditor.svelte:32`, a
  non-undefined fallback). Svelte 5 throws `props_invalid_value` exactly on `bind:`-ing `undefined` into
  a `$bindable` with a default.
- **Fix (one line, unbuilt, not applied here):** drop the `''` fallback → `value = $bindable()`, **or**
  pre-seed `draft_body[bcp]`/`draft_usage[bcp]` to `''` before the editor renders. Keep this as a **real
  P3 to fix** — it's in the shared editor every dict will use, not a throwaway preview.

### Waveform-decode — enrichment shipped, mystery solved (🟡 P3 → diagnosed transient-network)

Last run's carried Phase B item ("enrich the `{}` waveform log") **is done**: `Waveform.svelte:83` now
logs structured context. Today's 3 rows (all `dsuslak@iu.edu`, ayook dict, Windows/Firefox, one editing
session) carry:

```json
{ "name": "TypeError", "message": "NetworkError when attempting to fetch resource.",
  "mime": null, "bytes": null, "source": "url" }
```

So the waveform can't decode because the **audio-URL fetch itself failed** (`source:"url"`, transient
network) mid-editing-marathon — **not** a Safari-vs-Chrome codec issue as hypothesised. Playback (plain
`<audio>`) is unaffected → cosmetic. The enrichment did precisely its job. **Carried item CLOSED**; no
codec fix warranted. If it recurs at volume, the `source:"url"` NetworkError points at R2/CDN audio
delivery, not the decoder.
