### 🔴 P1 — Entry page `each_key_duplicate` crash

<Callout type="danger" title="Open on the current build — recommend a fix, do NOT auto-apply">
URL: <code>/sugtstun/entry/998c2a74-41a2-42c5-9607-27f82cc77060?q={"query":"anno","tags":["teglunaliq"]}</code> · app_version <code>1783303651101</code> (built <strong>2026-07-06 02:07:31Z</strong>, i.e. AFTER both entry-page fixes) · one user reopened the same entry 10–12× and it crashed every time.
</Callout>

**Root-cause hypothesis:** Svelte's keyed `{#each … (id)}` throws `each_key_duplicate` when two siblings share a key. The entry page keys children by id (`entry.senses (sense.id)`, `entry.audios (sound_file.id)`, `photos (photo.id)`, `sense.sentences (sentence.id)` in `EntryDisplay.svelte`/`EntryMedia.svelte`/`Sense.svelte`). A **duplicate junction row** in this user's *local* dict DB → two children with the same id → crash. This lines up with commit `24b080b1` ("harden dict junction sync ordering") — that stops *new* dupes but **does not heal already-corrupted local DBs**, so a pre-existing dupe still detonates the page. The follow-on `undefined is not an object (n().page / M(n).query=e / M(o).view=e)` errors are the entries-UI store unwinding after the crash, not separate bugs.

**Recommendation (not applied):** (a) **defensively de-dupe entry children by id** where the reactive rows are derived (belt-and-suspenders against corrupted local data), and/or (b) make the keyed `{#each}` blocks resilient so one bad row can't blank the whole page. File as an issue — see action items.
