## 1. Errors & crashes

### Headline: the `sync_failed` storm (see also `.issues/dict-sync-client-behind-storm-2026-07-05.md`)
`sync_failed` is a `warn`-level message, so it does **not** appear in the standard error-level cluster
table below вҖ” but at **9,862 of 23,523 total logs (41.9%)** and **89.7% of all warn rows (11,000)**
today, it's the single largest thing happening in the log stream and deserves top billing:

| kind | rows (24h) | notes |
|---|---|---|

| `snapshot_expired` | 113 | expected residual (see boot-health section) |
| `network` | 73 | normal connectivity blips |

**Root cause**: today's new dict-migration (`20260705_sentence_tokens_media_timings.sql`, part of the
`ea76229a` texts-pipeline commit, live from ~02:17 UTC) advanced the schema version every tab is
expected to declare. Any tab still on an older cached bundle now gets `409 schema_outdated` on every
sync attempt. Verified in code: `dict-sync-engine.ts`'s `sync_if_needed()` вҖ” called every 30s by the
engine's own timer вҖ” has **no blocked-state flag at all** (only checks `#in_flight`/`#stopped`), unlike
the admin engine's `blocked_by_client_behind`. `dict-instance.ts`'s `translate_sync_error()` only emits
a broadcast on `schema_outdated`; it never calls `engine.stop()` (contrast with the `storage_lost` path
in the same file, which explicitly stops the engine after `MAX_REOPEN_ATTEMPTS`). And
`sync-failure-classify.ts`'s `THROTTLED_KINDS` deliberately excludes `client_behind` (treated as a rare
hard failure that should "always ship") вҖ” so nothing caps the log volume either. There IS an
auto-reload recovery path (`recover_from_schema_outdated` in `[dictionaryId]/+layout.ts`), but it isn't
closing the loop for long-lived background tabs вҖ” Marlene Morales's tab has been on the **2026-07-03
build** for 36+ hours with **zero** other event types logged (no heartbeat/navigation вҖ” a fully
backgrounded tab), confirming the reload either isn't firing or isn't landing on a fresh bundle for
suspended tabs.

**Impact**: 26 distinct (user, dict) tab-pairs, 12 distinct users (11 named + anonymous spanning 34
dicts), including admin **Gregory Anderson** (98.5% of his 24h event count is this noise, not real
activity). No confirmed data loss today (affected named users besides Gregory logged no edit activity
in this window), but the mechanism means a stuck editor's writes wouldn't push until they reload вҖ” a
latent risk for the next occurrence. **Completely invisible on `/admin/analytics` and `/admin/health`**
вҖ” verified against `log-analytics.ts`: `top_events` only aggregates `level='info'` rows, and
`error_clusters`/`errors_by_version` only count `('error','unhandled_rejection','crash')` вҖ” `warn`-level
`sync_failed` hits neither. See Phase C for the panel proposal this grounds.
