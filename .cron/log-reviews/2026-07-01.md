### 🟡 Keyman `firstElementChild` on null (NEW recurring)
- **Where:** <File path="site/src/lib/components/keyboards/keyman/Keyman.svelte" line="86" /> —
  `inputEl = wrapperEl.firstElementChild`. `targetInput()` (and `waitForCKEditorToInitAndBeTargeted`)
  touch `wrapperEl` with **no null-check**; when the entry dialog is closed while Keyman's async init
  is still resolving, `wrapperEl` is `null` → `TypeError`.
- **Evidence:** 2× on 07-01 (03:22 on `/river/entries` after "Add Entry"→"Close"; 07:36 on a
  `/river/entry/…` after opening/closing an entry). Stack is the minified async-generator (Keyman
  loader). Same class the e2e harness already whitelists as ambient
  (<File path="site/tools/e2e/create-dict-soft-nav.mjs" line="25" /> — `/firstElementChild/i` in `AMBIENT`).
- **Recommended fix (don't apply):** early-return / optional-chain `wrapperEl` in both async fns
  (`if (!wrapperEl) return`). Cheap, kills a recurring unhandled_rejection now that a real editor is
  hammering entry dialogs. Low severity (the keyboard just fails to attach on a torn-down input).

### ⚪ Content-length 17 MB → `/api/auth/email/send-code`
- A 17,000,012-byte POST to the OTP `send-code` endpoint (`status 500`, `source='server'`), rejected
  by adapter-node's 16 MB `BODY_SIZE_LIMIT`. `send-code` takes a tiny JSON body, so this is a
  scanner/abuse probe, not a real flow. Correctly rejected — but logged at **`crash`**, inflating the
  crash count. **Noise candidate** (see Phase B).

### ⚪ `Could not load my dictionaries: 401`
- <File path="site/src/lib/dictionaries.ts" line="40" /> — the "my dictionaries" store `fetch('/api/me/dictionaries')`
  fires for an **anon** visitor (user_id null; breadcrumbs: opened `river` → navigated `/`), gets an
  expected 401, and `console.error`s it → logged at `error`. The 401 is correct behavior; the noise is
  ours. **Fix:** only fetch when signed in, or demote the log to `info`/`warn` on 401.
