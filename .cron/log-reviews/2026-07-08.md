## 5. Coverage gaps (Phase B)

Verified against the actual source — these are genuine blind spots, not already-logged paths.

1. **`/og` share-image endpoint has no error handling** — 🟠 highest-value gap.
   `site/src/routes/og/+server.ts` is a bare `JSON.parse(decode(...))` + `component_to_png(...)` with
   **no try/catch**. When the entry's `lh3` photo fetch fails (Satori/resvg), the whole render throws
   → **500**, and it surfaces as the *misleading* `og_font_unsupported` message.
   → Wrap in try/catch, render a **text-only fallback card** (never 500 a social-scraper request),
   and emit `log_server_event({ level:'warn', message:'og_render_failed', context:{ reason:
   'image_fetch'|'font'|'parse', dict_id } })`. **Why:** social/link-preview 500s are invisible to
   users but break shares, and today's mislabeled message actively misled triage.
2. **Error handler logs raw `Event` objects as `{"isTrusted":true}`** (54 rows) — when a caught value
   is a DOM `Event` (media/resource `error`), the logger stringifies it to nothing useful.
   → In `remote-log.ts`, when the value is an `Event`, extract `target.tagName` / `target.src|href` /
   `type` into `context`. **Why:** turns 54 blind rows into "audio 404 on `<audio src=…>`" etc.
3. **Client `Internal Error` 500-crashes (50) outnumber server crashes (9)** — handleError *does* log
   500s as `crash` (`hooks.server.ts:126`), so the gap is real: most of the 50 are **client-side
   stale-chunk dynamic-import failures** rendering SvelteKit's error page (no server call), not SSR
   500s. → Add a `reason` discriminator to the client crash log (`dynamic_import` vs `ssr_500`, both
   derivable client-side) so stale-chunk churn stops masquerading as server faults. Low priority
   (it's the stale-client story again) but ends the ambiguity.
4. **`vernacularName` SSR crash (2)** — an unguarded gloss-language lookup in a `.map` on some
   `+page` returns `undefined` → 500. All the reads I found are guarded (`?.`/`||`); one path isn't.
   → add the missing `?.`. Low freq (P3) but a hard 500.

*(Not gaps — verified well-instrumented already: `leader_boot_failed`, `sync_failed`,
`live_query_failed`, `initial dict sync failed` all carry rich `context` with `dict_id` / stage /
breadcrumbs. No new logging needed there.)*
