Browser wording differs, but the stack and route are identical:

## 5. Coverage gaps (Phase B)

1. **Search-worker boundary failure lacks operation context.** The patched global error capture found
   this regression, but each row only has the browser message and minified stack. When fixing the clone
   bug, add a narrow handled `search_worker_request_failed` error with `{ scope, dictionary_id,
   filter_keys, browser_message }`—never search text or filter values—so future worker-boundary faults
   identify the operation without session reconstruction. Current code has no such event.
2. **Import file PUT failure needs stage/status context.** `api/upload/+server.ts` already emits
   `upload_presign_failed` and `gcs_serving_url` already emits `gcs_serving_url_failed`; yesterday's
   generic “presign is invisible” suggestion is therefore already implemented and is not carried.
   Today's three failures happened in the direct XHR PUT, whose network-error branch logs only `Failed
   to upload file.` Enrich that client error with `{ stage:'put', status, file_type, bytes }`, excluding
   signed URLs and filenames. This would distinguish GCS rejection from offline/network interruption.
3. **No extra dirty-row logging needed.** The existing `dirty_rows_stuck` payload gave dictionary,
   row count, session, last sync, and last error—enough to make the `yokoim` problem actionable.
