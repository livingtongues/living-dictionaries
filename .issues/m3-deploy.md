## The deploy mechanism (RESEARCHED — corrects the standing assumption)

There is **NO GitHub Actions deploy workflow** in any LD branch (`main`, `dev`,
`svelte-5-migration` only have CI: lint/check/test/lighthouse). The "existing workflow that
auto-deploys" from the 2026-06-04 decision is actually the **vps-setup GitHub-webhook deploy**
used by poly/living/house:

1. Push to `livingtongues/living-dictionaries`, branch = `living.conf:DEPLOY_BRANCH`
   (**currently `svelte-5-migration`** — the example repo's branch is what's live on living now).
2. GitHub webhook → VPS `hooks.json` matches `refs/heads/${DEPLOY_BRANCH}` → runs `deploy.sh`.
3. `deploy.sh` (generated by `~/code/vps-setup/bin/sync` → `generate_deploy_sh`):
   - mints a GitHub App token, clones/`git reset --hard` `$BRANCH` into `/opt/hosting/sveltekit/code`
   - **preflight**: greps `$env/static/{private,public}` imports in `code/site/src`; every var
     must exist in `/opt/hosting/sveltekit/.env` or it aborts in <1s (before the 8-min build)
   - `cp /opt/hosting/sveltekit/.env code/.env`
   - `cd /opt/hosting/sveltekit && docker compose build && docker compose up -d`
4. `docker-compose.yml` (generated): `build: { context: ./code, dockerfile: Dockerfile }`;
   `env_file: .env`; volumes `/opt/hosting/data:/data` (+ kokoro, unused by LD); network `web`
   (external, shared w/ Caddy); loopback `127.0.0.1:3001:3000` for the host health-monitor.
5. **Dockerfile lives in the app repo root** (context `./code` = repo root). LD has none yet.

Caddy (also generated by bin/sync) already reverse-proxies `new.livingdictionaries.app` → the
`sveltekit` container on `web`. Behind Cloudflare ("full" SSL), Caddy uses `tls internal`.

## Findings from the dep/env audit

- **adapter-node build is clean (exit 0).** AWS SDK v3 (`@aws-sdk/client-s3`, `client-ses`,
  `s3-request-presigner`) sits in **devDependencies** and **bundles fine** (only harmless
  `Circular dependency` warnings) — no need to move it to `dependencies`. The example keeps the
  same trio in `dependencies` (externalized) but bundling works for us, so leave as-is.
- **Native/runtime deps already correctly in `dependencies`:** `better-sqlite3`, `@resvg/resvg-js`,
  `satori`, `satori-html`, `jose`, `drizzle-orm`, `wa-sqlite`. ✅ (M0.5 audit holds.)
- **`better-sqlite3` build:** root `package.json` has `pnpm.onlyBuiltDependencies:
  ["better-sqlite3"]` — gates pnpm v10 so the native compile runs. Dockerfile builder installs
  WITHOUT `--ignore-scripts` so the `install` hook compiles; runner copies the `.node` binary.
- **`svelte-look` `link:../../svelte-look`** points outside the repo — but house+example carry the
  identical link and deploy fine: pnpm just creates a dangling symlink; the prod build never
  imports it (stories/variants only). No action.
- **Build-time STATIC env vars** (preflight-checked, must be in VPS `.env`):
  - `$env/static/public`: `PUBLIC_mapboxAccessToken`, `PUBLIC_STORAGE_BUCKET`
  - `$env/static/private`: `AWS_SES_ACCESS_KEY_ID`, `AWS_SES_REGION`, `AWS_SES_SECRET_ACCESS_KEY`
    (via `send-email.ts`, reachable from `/api/auth/email/send-code` + 9 `/api/email/*` routes).
    ⚠ The example uses **nodemailer**, not AWS SES — so living's current `.env` likely lacks the
    `AWS_SES_*` keys. Jacob must add them (real → login/notify emails send; placeholder → build
    passes but email 500s).
- **Runtime DYNAMIC env vars** (`$env/dynamic/private`, read per-request): `JWT_SECRET`,
  `PUBLIC_GOOGLE_OAUTH_CLIENT_ID`, `GCLOUD_MEDIA_BUCKET_ACCESS_KEY_ID`,
  `GCLOUD_MEDIA_BUCKET_SECRET_ACCESS_KEY`, `PROCESS_IMAGE_URL`, `DATA_DIR` (=`/data`).
- Stale stub refs (`PUBLIC_SUPABASE_*`, `SUPABASE_SERVICE_ROLE_KEY`) are read via dynamic/public
  or unused — undefined is harmless; not required.
